Technical End-to-End System Flow Guide

1. Member Onboarding & KYC Lifecycle

This process transitions a user from initial registration to a fully verified stakeholder capable of earning points.

Step Application Action Primary Table Mutations Secondary Table / Events
1. Registration Mobile App Signup users: Insert record. block_status = 'basic_registration'. retailers/electricians: Insert profile record with user_id.
2. KYC Upload Stakeholder App kyc_documents: Insert with status = 'pending'. audit_logs: Log submission metadata.
3. Admin Review updateKycDocumentStatusAction kyc_documents: status -> 'verified', verified_at = NOW(). Event: USER_KYC_APPROVED emitted.
4. Activation System Handler users.approval_status_id: Update to 'Active'. inapp_notifications: Welcome alert created.

2. Point Accrual (The Scanning Journey)

A. QR Validation & Rule Lookup

When a QR code is scanned, the system performs a multi-stage lookup:

  • Integrity Check: Queries qr_codes to ensure is_scanned is false.
  • Base Configuration: Fetches points_per_unit from sku_point_config based on user_type_id and sku_variant_id.
  • Rule Overlays: Applies sku_point_rules (e.g., multiplier rules for specific regions or timeframes).

B. Settlement & Accounting

  • Logging: Inserts into retailer_transaction_logs with state PENDING.
  • Approval: Upon approval, the member's points_balance is updated in the profile table (e.g., retailers).
  • Auditing: A record is inserted into retailer_ledger to track the opening_balance and closing_balance of the transaction.

3. Redemption & Payout Lifecycle

The conversion of virtual points into UPI, Bank, or Amazon rewards.

  1. Request Submission: Points are validated; if sufficient, they are deducted from points_balance and added to total_redeemed.
  2. Ledgering: A DEBIT entry is created in the stakeholder's ledger.
  3. Redemption Record: A row is added to redemptions and either redemption_upi or redemption_bank_transfers.
  4. Payout Execution: Admin approval triggers the REDEMPTION_APPROVED handler, which initiates a Razorpay Payout.
  5. Finality: status is updated to SUCCESS and the utr is saved upon webhook receipt.

4. Replication Guide for Fresh Programs

Seeding Priority: When starting a fresh program, you MUST seed tables in this order to respect Foreign Key constraints:

INSERT INTO approval_statuses (name) VALUES ('KYC_PENDING'), ('ACTIVE');

-- 2. Role Setup
INSERT INTO user_type_level_master (level_name) VALUES ('Role');

-- 3. SKU Hierarchy

5. System Monitoring

  • API Health: Monitor third_party_api_logs for failures in external communications (Razorpay, Amazon).
  • Security: Check audit_logs for any unauthorized manual point adjustments.
  • Performance: Aggregated data for the dashboard is derived from the various transactions and transaction_logs tables.
